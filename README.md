# Conversant Monorepo

Conversant is a demo voice conversation application built as a monorepo.

## Development Note

- This codebase was developed with AI coding agents and manually reviewed/refined.

## Architecture Status

- Current implementation uses REST endpoints for STT/LLM/TTS and turn orchestration.
- This is intentionally simple for the current demo stage.
- This approach is not optimal for realtime scale.
- A production-grade next iteration should move to a stateful transport such as WebSockets (or WebRTC where appropriate).

## Stack

- Turborepo.
- Yarn Berry (PnP).
- Next.js App Router (`apps/frontend`).
- Prisma + PostgreSQL.
- shadcn/ui components.
- Biome (no ESLint/Prettier).

## Workspace Layout

- `apps/frontend`: UI and server routes.
- `packages/backend-core`: request handlers for STT/LLM/TTS/session flows.
- `packages/backend-data`: Prisma schema, migrations, auth/invite/session persistence, conversation persistence.
- `packages/config`: shared runtime env readers.
- `packages/conversation-engine`: frontend conversation engine package.
- `tools/invite`: standalone CLI package for invite generation.

## Quick Start

Run from repository root (`code`):

```bash
yarn install
cp .env.example .env
docker compose up -d
yarn db:migrate:deploy
yarn workspace frontend dev
```

Open `http://localhost:3000`.

## Common Commands

- `yarn dev`: start workspace dev tasks.
- `yarn build`: build all workspaces.
- `yarn lint`: run Biome checks through Turbo.
- `yarn format`: run Biome formatter through Turbo.
- `yarn check-types`: run TypeScript checks through Turbo.
- `yarn test`: run tests through Turbo.
- `yarn db:migrate:deploy`: apply Prisma migrations.
- `yarn invite:generate -- --help`: show invite CLI help.

## Environment Variables

Copy `.env.example` to `.env` and set at least:

- `DATABASE_URL`: PostgreSQL connection string.
- `OPENAI_API_KEY`: API key for OpenAI-compatible provider.
- `INVITE_ADMIN_SECRET`: secret for hashing and validating invite tokens.
- `SESSION_SECRET`: secret for hashing session tokens.

Frequently used optional variables:

- `OPENAI_BASE_URL`: custom OpenAI-compatible base URL.
- `SESSION_TTL_HOURS`: session lifetime.
- `INVITE_BASE_URL`: base URL used by invite generator output.
- `ALLOW_PUBLIC_ACCESS`: bypass invite/session checks when `true`.
- `CONVERSATION_MAX_DURATION_SEC`: default conversation time limit.
- `ASSISTANT_SYSTEM_PROMPT`: baseline system prompt.
- `ASSISTANT_MAX_OUTPUT_TOKENS`: response length control.

## Auth Approach

- Authentication in this demo is based on invite URLs.
- This is an intentional simplification to avoid introducing a full auth provider and extra auth flows at the current stage.

## Invite Mechanism

- Invite links are generated by CLI and consumed via `/api/auth/invite/consume?token=...`.
- Each invite supports:
- Time to first use (`--ttl-hours` or `--ttl-days`).
- Maximum allowed consumes (`--max-uses`).
- Optional per-invite conversation limit (`--conversation-max-duration-sec`).
- On successful consume, server creates a session cookie.
- In private mode (`ALLOW_PUBLIC_ACCESS=false`), root page access is single-entry per created session and then requires a new invite/session.
- In public mode (`ALLOW_PUBLIC_ACCESS=true`), invite/session checks are bypassed.

Examples:

```bash
# 10 uses, 1 year to first use
yarn invite:generate -- --ttl-days 365 --max-uses 10 --base-url http://localhost:3000

# 3 uses, custom per-session conversation limit from invite
yarn invite:generate -- --ttl-hours 24 --max-uses 3 --conversation-max-duration-sec 180 --base-url http://localhost:3000
```

## Before Publishing

- Verify `.env` is not committed.
- Rotate keys if any test keys were exposed.
- Ensure `ALLOW_PUBLIC_ACCESS` matches intended deployment mode.
- Run `yarn lint`, `yarn check-types`, and `yarn build`.
